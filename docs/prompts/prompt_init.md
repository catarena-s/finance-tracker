Проект: Full-Stack Трекер личных финансов  

Цель: Сгенерировать полный scaffold full-stack приложения с нуля (frontend, backend, БД), начиная с **правил работы IDE**, **архитектуры** и **плана разработки**, с фиксацией всех шагов в `ARCITECTURE.md` и отчётов в `REPORT.md`.

Результаты:
- Работающее приложение с минимум 4-5 ключевыми фичами 
- Тесты, документация и CI pipeline 
- Демо-развёртывание (Docker-compose, локально)

Технологии: 
- React / Next.js или Vue,Python backend 
- PostgreSQL 
- Docker, GitHub Actions
 
Frontend:
- React / Next.js или Vue  
- Дашборд с графиками и статистикой  
- Пагинация списков  
- Адаптивная вёрстка для desktop и mobile  

---
Функциональные требования (Functional Requirements):
- CRUD основных сущностей с валидацией на клиенте и сервере 
- Поиск и фильтрация по ключевым полям 
- Дашборд с визуализацией данных (графики / статистика) 
- Пагинация списков 
- Адаптивная вёрстка (desktop + mobile) 
- Транзакции: сумма, дата, категория, описание, тип (доход/расход) 
- Справочник категорий (иконки, цвета) с возможностью CRUD 
- Месячные бюджеты: установка лимита на категорию, прогресс-бар расхода 
- Дашборд: круговая диаграмма расходов по категориям, линейный график динамики за 6 месяцев, топ-5 категорий 
- Фильтрация транзакций по дате, категории, сумме (от/до) 
- Импорт транзакций из CSV-файла (маппинг колонок при загрузке) 
- Экспорт отфильтрованных транзакций в CSV 
- Повторяющиеся транзакции (ежемесячные подписки, зарплата) 
— автосоздание по расписанию 
- Мультивалютность: выбор основной валюты, хранение курса на момент транзакции 

---
Нефункциональные требования (Non-functional Requirements):
- Seed-данные: приложение запускается с реалистичным набором демо‑данных 
- REST API с OpenAPI/Swagger документацией 
- Не менее 10 unit/integration тестов 
- CI pipeline (GitHub Actions): lint + тесты (возможно, сборка Docker-образа) 
- Docker Compose: backend + frontend + БД 
— запуск одной командой - файл README инструкцией по запуску 
- Файл ARCITECTURE с описанием архитектуры и плана разработки (начните с него) 
- (Опционально) Ссылка на видео-демонстрацию работы приложения может упростить и ускорить проверку 
- Файл REPORT с вашим описанием истории создания проекта, с какими ключевыми проблемами вы столкнулись, как решали. Ваши удачные и неудачные идеи промтов. Добавляйте туда данные по мере работы, не в самом конце. 
- Проект создан в новом, пустом GIT репозитории. Существует адекватная история коммитов (не всё разом).
**Seed-данные:** 200+ транзакций за 6 месяцев, 12 категорий, 3 бюджета

Критерии оценки 
- Выполнены все **общие требования** (CRUD, поиск, дашборд, пагинация, адаптивная вёрстка, тесты, CI, Docker Compose, OpenAPI, README) 
- Выполнены все **функциональные требования выбранного варианта** 
- Приложение поставляется с **seed-данными** указанного объёма

Каждый ключевой этап должен быть оформлен в отдельную ветку, шаги внутри этапа должны комититься в эту ветку, по завершении этапа должен быть сформирован PR в main.

Последовательность разработки должна быть такая: 
- комиты делаются в указанный репозиторий от имени автора(автор не должен сохраняться нигде в документации в репозитории)
- определяются правила разработки, комитов и др для ide - комитится
- определяется архитектура проекта и документируется(ARCITECTURE.md) - комитится
- потом составляется план разработки, фиксируется в ARCITECTURE.md - сохраняется, комитится 
- начинается разработка по составленному плану. 
- После разработки каждой фичи перед комитом в плане отмечается выполненный  завершенный этап.

Кодирование и стиль
- Код должен быть чистым, читаемым и модульным  
- Backend: REST API с понятными методами и DTO  
- Frontend: компоненты небольшие, переиспользуемые, именование по конвенции выбранного фреймворка  
- Все новые функции должны быть покрыты комментариями в формате JSDoc / Python docstring  

Тесты
- Unit-тесты для каждого модуля  
- Integration-тесты для взаимодействия backend-frontend  
- Минимум 10 тестов на ключевые фичи  
- Тесты пишутся **параллельно с реализацией функционала**  

Документация
- Каждый новый API-эндпоинт документируется через OpenAPI/Swagger  
- README.md: инструкция по запуску и использованию приложения, генерации seed-данных, docker-compose  
- REPORT.md ведётся по мере работы: фиксируем проблемы, решения, удачные и неудачные идеи промтов  

Коммит/PR
- Коммиты делаем только после **завершения шага разработки и тестирования**  
- PR создаём после завершения этапа  
- Сообщения коммитов: `[Этап: <название>] Выполнен шаг: <описание>`  

CI/CD
Продумать каие Actions будут выполнять на обычные комиты в рабочую ветку, а какие после приняти PR в main
- GitHub Actions:  
- lint + форматирование  
- unit/integration тесты  
- (опционально) сборка Docker-образов  
- Docker Compose:  
- backend + frontend + БД  
- запуск одной командой: `docker compose up`

Выполнены все функциональные требования выбранного варианта.
